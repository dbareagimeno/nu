# Incremental implementation plan

Status: draft. This document orders the construction of `enu` into a sequence
of **sessions, each with a single deliverable feature**. It is not a contract
(it does not freeze the API; that's [api.md](api.md)) but a construction map:
in what order the kernel is built, why that order, and what makes a session
*done*.

## Tracking: where to resume in each session

Each session starts with a blank context in an ephemeral container, so
**progress state lives in the repository, not in Claude's memory**. The
mechanism is the same pattern already used by `problemas.md` (*status* field)
and `adr.md`: a visible pointer, an append-only logbook, and the discipline of
updating them in the feature's own commit. Three sources, redundant on
purpose:

## How to use this plan

Each row in the tables below is **one Claude session**: a unit of
work that fits in a context window, leaves the tree compiling, and adds an
*observable and tested* capability. The granularity is deliberate: one session
≈ one API submodule (sometimes a slice of one, when the submodule is large).

### Session contract (Definition of Done)

A session is not done until it meets all five:

1. **Compiles**: `go build ./...` green, `CGO_ENABLED=0` (ADR-001).
2. **Is tested at the level its logic demands** (see "Test policy"): *every*
   session carries at least one Lua snippet that exercises the signature from
   the extension author's side (the harness arrives in S02). Sessions with
   **key logic** (marked 🔒 in the inventory) **additionally** carry exhaustive
   Go unit tests of their edge cases — those are **never skipped**. A thin
   wrapper over the stdlib doesn't invent tests for someone else's code: the
   snippet and the phase checkpoint cover it.
3. **Respects the spec**: the implemented signature and semantics are those of
   `api.md` §corresponding, **⏸**/**[W]** markers included. Not one function more.
4. **Doesn't regress**: features from previous sessions stay green.
5. **Leaves a trace**: commit in Spanish citing the session; if there was a finding,
   linked from `problemas.md`.

### Test policy: what deserves a unit test

Not everything deserves a unit test, but **key logic can't be skipped**.
The rule isn't "coverage" but *where the risk is*. A Go unit test
(table-driven, with its edge cases) is **mandatory** when any of the
following holds:

- **The logic is ours**: an algorithm, a state machine, or an invariant
  that we wrote ourselves — not a delegation to Go's stdlib or a
  library. A cancellation semantics, a parser, a viewport clip.
- **The failure is silent or edge-case**: off-by-one, ordering, concurrency,
  clipping, incremental parsing, EOF, backpressure. Things a happy path
  (the snippet or the checkpoint) **doesn't** touch.
- **It implements a `G##` finding**: each G## encodes an edge case that took
  effort to decide. Its regression must be shielded by a test that **names it**
  (`// G27: out[i] aligned with fns[i]`), so no one undoes it unknowingly.

**The Lua snippet + phase checkpoint are enough** (don't invent a unit test) when:

- It's a **thin wrapper** over Go's stdlib or a library (`toml`/`yaml`
  encode, `sys.platform/hostname/now_ms`, `fs.cwd`): testing that is testing someone
  else's code.
- It's **pass-through glue** with no decision of its own (most getters).
- It's **visual/interactive rendering**: it's validated by inspecting the resulting
  `Block` or looking at the screen in the checkpoint, not pixel by pixel.

## Why this order (the dependency graph)

The kernel isn't built module by module in the order `api.md`
enumerates them, but by **execution dependency**:

1. **The scheduler is the keel.** Almost the whole API is **⏸**: you can't
   test `enu.fs.read` without a task to await it or a loop to resume it. That's
   why, after the skeleton, the first and most expensive thing is the Lua
   coroutine ↔ Go goroutine bridge with its event loop (ADR-004). Everything hangs off that.
2. **Events and loader** come early because they're the substrate for *loading and
   testing* anything real (`init.lua`, `core:ready`) and for the robustness
   guards (`pcall` per boundary, ADR-008).
3. **IO, network, text, search** are relatively independent Go primitives
   from one another once the scheduler exists; they're ordered by unblocking value
   (`fs`/`proc` before `http`, because network tests rely on files).
4. **The UI comes after a veto spike** (ADR-007, open question #1 of
   [arquitectura.md](arquitectura.md)): the compositor + toolkit-in-Lua
   architecture isn't committed to without first proving it's fluid.
5. **Workers** arrive late because they're opt-in parallelism: nothing on the
   basic path depends on them, and they need the **[W]** API to already exist to
   trim it with `caps`.
6. **The official extensions** (agent, providers, MCP, chat, toolkit, repl)
   are the last phase: they're Lua on top of the now-frozen API, with no kernel
   privilege (ADR-003). Their internal order is dictated by their own coupling.

## Coherence with the design flow

This plan **doesn't replace** the `problemas.md`/`adr.md`/`pospuesto.md` flow:
it consumes it. If a session uncovers a crack, it's opened as a `G##`, resolved in
*all* the affected documents, and only then implemented. If a session
makes a new decision (e.g., the outcome of spike S28), it's logged as a new
ADR, never rewriting an old one. Code is the last link in the
coherence chain, not the first.
